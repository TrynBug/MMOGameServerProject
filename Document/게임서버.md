# 클라이언트
- 트리 오브 세이비어 게임을 모티브로 합니다.
- Unity 클라이언트 입니다.
- 3D 모델링을 사용하며, 쿼터뷰(Quarter View)를 사용합니다.
- 좌표계는 직교좌표계 입니다.
- 캐릭터는 클릭 투 무브(Click-to-Move) 방식으로 움직입니다. 마우스 좌클릭으로 땅을 찍으면 캐릭터가 해당 위치로 이동합니다.  
- 스킬은 키보드로 사용합니다.

## 캐릭터가 이동할 때
- 클라가 먼저 경로를 계산해서 캐릭터 이동을 시작합니다(빠른 반응성). 그리고 목적지를 서버에 전송합니다.
- 서버도 같은 경로를 계산해서 시뮬레이션 합니다.
- 서버는 출발지점, 경로, 목적지를 검사하여 이동할 수 없거나 부정한 이동이면 캐릭터의 경로를 다시 지정하여 클라이언트에게 전달합니다.

# 서버 tick
서버는 컨텐츠 스레드를 업데이트하여 서버 로직을 업데이트 합니다.  
1개 서버에 여러개의 컨텐츠 스레드가 있고, 1개의 컨텐츠 스레드는 여러개의 컨텐츠(Stage, 던전 등)를 소유합니다.  
1개 컨텐츠는 1개 컨텐츠 스레드에 소유되어 있어서 반드시 1개 스레드에서 처리됩니다.  
1개 컨텐츠는 여러개의 오브젝트(캐릭터, 몬스터, 프랍 등)를 소유합니다.  

서버의 tick은 단계별로 나누어져 있습니다.  
유저(캐릭터), 중요한몬스터(보스 등) 등의 중요한 오브젝트는 50ms 마다 업데이트 됩니다.  
일반 몬스터, 프랍 등의 중요하지 않은 오브젝트는 200ms 마다 업데이트 됩니다.  
그 외에 100ms, 500ms 등의 단계를 더 넣을수도 있습니다(각 단계는 50ms의 배수이어야 함).  
그래서 컨텐츠 스레드는 50ms 마다 업데이트 되어야 하며, 50ms 마다 업데이트해야할 tick 단계를 업데이트 합니다.  
오브젝트는 생성된 다음 컨텐츠에 소유될 때 어떤 단계의 tick 마다 업데이트 될지를 결정해야 합니다.  


# 게임서버
- Stage 클래스
	- Stage는 캐릭터가 돌아다닐 수 있는 1개의 필드를 가집니다.
	- 필드는 섹터 단위로 나누어져 있습니다.
	- 1개 Stage는 단일 스레드에서 처리됩니다.
	- Stage는 시스템 메시지큐를 가집니다. 이 메시지큐는 유저입장, 유저연결끊김, 서버종료 등의 시스템 메시지를 처리하기 위한 큐입니다.
	- Stage가 업데이트될 때 시스템 메시지큐의 메시지를 모두 처리합니다.
	- Stage는 자신 Stage에 속한 유저객체를 소유합니다.
	- Stage가 업데이트될 때 유저객체도 함께 업데이트합니다.
	
- 오픈필드 클래스
	- 오픈필드는 Stage를 상속받습니다. 그래서 단일스레드에서 처리됩니다.
	- 게임서버당 1개의 오픈필드가 존재합니다.
	- 오픈필드는 게임서버가 시작될 때 생성되고, 1개의 컨텐츠 스레드에 배정됩니다. 오픈필드를 소유한 컨텐츠 스레드는 오픈필드 전용 스레드가 되어서 다른 Stage를 소유할 수 없습니다.
	
- 공용던전 클래스
	- 공용던전은 Stage를 상속받습니다. 그래서 단일스레드에서 처리됩니다.
	- 게임서버가 생성하는 던전으로, 시작 시간과 종료 시간이 있습니다. 유저는 이 시간 사이에만 입장할 수 있습니다.
	
- 유저던전 클래스
	- 유저던전은 Stage를 상속받습니다. 그래서 단일스레드에서 처리됩니다.
	- 1명의 유저가 생성한 던전입니다.
	- 던전을 생성한 유저가 유저던전을 닫거나, 1시간이 지나면 던전이 닫힙니다.
	
- 컨텐츠 스레드
	- 오픈필드 전용 컨텐츠 스레드는 1개 입니다. 이 스레드는 오픈필드를 소유하며 다른 Stage 객체를 소유할 수 없습니다.
	- 일반 컨텐츠 스레드의 수는 `서버 CPU 코어 개수 - 1` 입니다.
	- 공용던전, 유저던전은 생성될 때 자신이 배정될 컨텐츠 스레드 번호를 다음 연산으로 찾습니다: `컨텐츠스레드번호 = 자신의ID mod 일반컨텐츠스레드수`
	
- 유저 클래스
	- 유저 1명을 나타냅니다.
	- 유저는 1개의 Stage 객체에 소유됩니다.
	- 유저는 Stage 객체가 업데이트될 때 업데이트 됩니다.
	- 유저객체에는 유저가 접속한 게이트웨이서버ID가 저장되어 있습니다. 유저에게 패킷을 보내려면 해당 게이트웨이 서버에 보냅니다.
	
- 유저가 동일 게임서버 내에서 Stage를 이동할 때
	- Stage 이동요청 패킷에 대해 클라이언트와의 규칙을 정합니다: 클라이언트는 Stage 이동요청 패킷을 보낸 다음에는 응답을 받기 전까지 다른 패킷을 보내지 않는다.
	- 그러면 Stage 이동요청에 대한 처리 흐름은 다음과 같습니다:
		1. 클라이언트로부터 Stage 이동요청 패킷을 받았습니다. 그러면 유저객체의 메시지큐에 패킷이 삽입됩니다.
		2. Stage가 업데이트될 때 유저객체의 패킷이 순서대로 처리됩니다. 만약 Stage 이동요청 패킷이 확인되었다면 이동을 시작합니다. 그리고 나머지 패킷들은 모두 버립니다. 이동이 완료되기 전까지는 모든 패킷을 버립니다.
		3. 유저객체를 현재 Stage에서 제거하고, 목적지 Stage의 메시지큐에 `유저입장` 시스템 메시지를 삽입합니다.
		4. 다음번에 목적지 Stage가 업데이트될 때 `유저입장` 시스템 메시지를 처리하여 유저를 입장시킵니다.
		5. 목적지 Stage가 유저에게 Stage 이동완료 패킷을 보냅니다.
	
- 유저가 다른 게임서버의 Stage로 이동할 때
	- 다른 게임서버의 Stage로 이동하는 경우는 클라이언트가 친구나 파티원이 있는 위치로 이동요청을 하는 경우입니다.
	- 다른 게임서버의 Stage로 이동요청 패킷에 대해 클라이언트와의 규칙을 정합니다: 클라이언트는 Stage 이동요청 패킷을 보낸 다음에는 응답을 받기 전까지 다른 패킷을 보내지 않는다.
	- 그러면 Stage 이동요청에 대한 처리 흐름은 다음과 같습니다:
		1. 클라이언트로부터 다른게임서버의 Stage로 이동요청 패킷을 받았습니다. 그러면 유저객체의 메시지큐에 패킷이 삽입됩니다.
		2. Stage가 업데이트될 때 유저객체의 패킷이 순서대로 처리됩니다. 만약 다른게임서버의 Stage로 이동요청 패킷이 확인되었다면 이동을 시작합니다. 그리고 나머지 패킷들은 모두 버립니다. 이동이 완료되기 전까지는 모든 패킷을 버립니다.
		3. 유저객체를 현재 Stage에서 제거하고, 유저데이터를 게임DB에 저장합니다.
		4. 게임DB에 저장이 완료되면 게이트웨이 서버에 `목적지 게임서버ID`를 전송합니다.
		5. 게이트웨이 서버는 유저의 연결을 목적지 게임서버로 변경하고, 목적지 게임서버에 유저 접속을 알립니다.
		6. 목적지 게임서버는 게임DB에서 유저 정보를 다시 로드해서 유저객체를 생성합니다.
		7. 목적지 게임서버는 유저객체를 적절한 Stage에 배정하고, 유저에게 Stage 이동완료 패킷을 보냅니다.	
	- 목적지 게임서버에 입장하려다가 Stage를 찾지 못해서 이동에 실패한 경우
		- 목적지 게임서버의 오픈필드로 이동시킵니다.
	- 게이트웨이서버가 목적지 게임서버를 찾지 못해 이동에 실패한 경우
		- 원래 게임서버의 오픈필드로 이동시킵니다.
	
- 게임서버가 게이트웨이 서버에 패킷전송을 요청할 때
	- 유저객체에 게이트웨이서버ID가 저장되어 있기 때문에 게임서버는 유저가 어떤 게이트웨이서버에 접속해있는지 알수있습니다. 이 게이트웨이서버에 패킷전송을 요청합니다.
	- 브로드캐스팅 할 때는 게임서버가 `게이트웨이서버ID : 유저ID목록` 으로 그룹화하여 유저가 접속한 게이트웨이 서버에만 유저ID목록을 전달합니다.

- 게임DB에 접근
	- 게임DB에서 유저데이터 읽는 시점
		- 유저가 게임서버에 입장할 때 게임DB에서 유저데이터를 읽어 유저객체를 생성합니다.
	- 게임DB에 유저데이터 업데이트 하는시점
		- 유저가 게임서버에서 나갈 때
		- 유저가 Stage를 이동할 때 
		- 던전 클리어 등의 중요 이벤트 발생 때
		- 마지막으로 업데이트한 지 3분이 지났을 때
