# 라이브러리 관리 방식
- 솔루션 내부에서 빌드되는 라이브러리 파일은 $(SolutionDir)OUTPUT/$(Configuration)/ 에 생성됩니다. 다른 프로젝트들은 이 폴더를 참조합니다. (프로젝트 속성 -> VC++ 디렉터리 -> 포함 디렉터리) 
- 솔루션 내부에서 빌드되는 라이브러리의 include 헤더파일은 각각의 라이브러리 프로젝트 안에 그대로 있습니다. 다른 프로젝트들은 각각 라이브러리 프로젝트의 폴더를 참조하여 사용합니다. (프로젝트 속성 -> VC++ 디렉터리 -> 라이브러리 디렉터리)
- vcpkg로 설치되는 라이브러리(protobuf 등) 파일은 $(SolutionDir)vcpkg/x64-windows-static-md-2022/lib/$(Configuration) 에 있습니다. 다른 프로젝트들은 이 폴더를 참조합니다. (프로젝트 속성 -> VC++ 디렉터리 -> 포함 디렉터리)
- vcpkg로 설치되는 라이브러리의 include 헤더파일은 $(SolutionDir)vcpkg/x64-windows-static-md-2022/include/ 에 있습니다. 다른 프로젝트들은 이 폴더를 참조합니다. (프로젝트 속성 -> VC++ 디렉터리 -> 라이브러리 디렉터리) 
- 라이브러리 링크는 프로젝트 속성에서 수동으로 입력합니다. (프로젝트 속성 -> 링커 -> 입력 -> 추가 종속성)
- protobuf와 관련된 라이브러리는 숫자가 너무 많기 때문에 .props 파일로 따로 관리하고 있습니다.
	- $(SolutionDir)Common 폴더에 ProjectPropertyMacro.props 파일이 있는데, 여기에 ProtobufLibs, ProtobufLibsDebug 매크로가 정의되어 있습니다.
	- 프로젝트 속성에서 $(ProtobufLibs), $(ProtobufLibsDebug) 매크로로 링크할 라이브러리를 입력합니다. (프로젝트 속성 -> 링커 -> 입력 -> 추가 종속성)
	- (visual studio에서 프로젝트에 .props 파일 추가하는방법: 보기 -> 다른창 -> 속성관리자 -> 프로젝트 우클릭 -> 기존 속성시트 추가

visual studio에서 디버깅 할때도 $(SolutionDir)OUTPUT/$(Configuration)/ 에 있는것을 사용합니다. (프로젝트속성->디버깅->작업디렉터리 설정해놔서 이렇게 작동함)

# 패킷
모든 패킷은 PacketGenerator 프로젝트가 protoc 로 생성합니다.  
패킷 ID는 PacketGenerator 프로젝트의 packet_id.proto 를 참조하면 됩니다.  
유저와 서버가 주고받는 패킷ID는 enum GamePacketId 에 있습니다.  
서버와 서버가 주고받는 패킷ID는 enum ServerPacketId 에 있습니다.   

# 서버 간단 설명
- 레지스트리 서버
	- 모든 서버의 연결상태를 관리하는 서버입니다.
	- 모든 서버는 레지스트리 서버로부터 다른 서버의 정보를 얻습니다.
	- 레지스트리 서버는 1개만 존재합니다.
	- 모든 서버들은 가동되었을 때 레지스트리 서버에 연결하여 자신의 정보를 등록합니다.
	- RegistryDB와 연결됩니다.

- 로그인 서버
	- 클라이언트가 최초 접속하는 서버입니다.
	- 클라이언트는 로그인 서버로부터 접속가능한 게이트웨이 서버의 IP를 전달받아 게이트웨이 서버로 접속합니다. 그래서 로그인서버는 모든 게이트웨이서버에 접속합니다.
	- 로그인서버는 1개만 존재합니다.
	- AccountDB와 연결됩니다.

- 게이트웨이 서버
	- 클라이언트가 게임플레이를 위해 접속하는 서버입니다.
	- 서버는 N대가 있으며 실시간으로 개수를 늘리거나 줄일 수 있습니다.
	- 대규모 MMORPG에서 브로드캐스팅 부하를 줄이기 위해 존재합니다.
	- 클라이언트로부터 받은 패킷을 그대로 게임서버에 전달합니다. 게임서버가 게이트웨이서버에 접속하기 때문에 이 연결을 통해 게임서버로 전달합니다.
	- 게임서버로부터 패킷과 대상유저 정보를 전달받아서 대상유저 모두에게 패킷을 브로드캐스팅 합니다. 
	
- 게임 서버
	- 게임로직을 돌립니다.
	- M대가 있으며 실시간으로 개수를 늘리거나 줄일 수 있습니다.
	- 모든 게이트웨이 서버에 접속합니다. 클라이언트와 직접적인 연결은 없습니다. 게이트웨이 서버는 N대가 있습니다. 
	- 패킷을 보낼때는 게이트웨이 서버에게 패킷과 보낼대상유저 목록을 보내면 게이트웨이 서버가 클라이언트에게 전달해줍니다.
	- GameDB와 연결됩니다.
	
- 채팅 서버
	- 채팅만을 담당합니다.
	- 채팅서버는 1대만 존재합니다.
	- 모든 게임서버에 접속합니다.
	- 게임서버의 채팅 요청을 받아서, 채팅 보낼 대상이 있는 다른 게임서버들에게 브로드캐스팅을 해줍니다.

# 서버 연결 토폴로지

## 연결 방향 규칙
- **연결 방향 표기**: "A → B" 는 A가 connect 측(Connector), B가 listen 측(Listener) 임을 의미합니다.
- **클라이언트(Client)** 는 외부 유저(Unity)를 의미합니다. 클라이언트는 항상 Connector 입니다.
- **레지스트리 서버는 모든 서버 연결의 허브가 아닙니다**. 단지 서버 정보를 중계하는 역할만 합니다. 실제 서버간 데이터 통신은 레지스트리를 거치지 않습니다.
- **포트 분리 정책**: 클라이언트용 포트와 서버 내부 통신용 포트를 분리합니다. ServerBase에 클라이언트용 NetServer와 내부서버용 NetServer가 구분되어 있습니다.

## 전체 연결 토폴로지 표

| Connector (연결하는 쪽) | Listener (받는 쪽) | 연결 개수      | 연결 시점                          | 용도                                         |
| ----------------------- | ------------------ | -------------- | ---------------------------------- | -------------------------------------------- |
| 클라이언트              | 로그인 서버        | 1 → 1          | 클라이언트 최초 접속               | ID/Password 인증, 인증토큰 발급              |
| 클라이언트              | 게이트웨이 서버    | 1 → 1          | 로그인 성공 후                     | 게임플레이용 메인 통신                       |
| 로그인 서버             | 레지스트리 서버    | 1 → 1          | 로그인 서버 시작 시                | 자기 등록, 게이트웨이 정보 폴링              |
| 게이트웨이 서버         | 레지스트리 서버    | N → 1          | 게이트웨이 서버 시작 시            | 자기 등록, 로그인/게임서버 정보 폴링         |
| 게임 서버               | 레지스트리 서버    | M → 1          | 게임 서버 시작 시                  | 자기 등록, 게이트웨이 정보 폴링              |
| 채팅 서버               | 레지스트리 서버    | 1 → 1          | 채팅 서버 시작 시                  | 자기 등록, 게임서버 정보 폴링                |
| 로그인 서버             | 게이트웨이 서버    | 1 → N (모든쌍) | 로그인서버가 게이트웨이 정보 수신 후 | 클라이언트 인증토큰 사전 등록                |
| 게임 서버               | 게이트웨이 서버    | M → N (모든쌍) | 게임서버가 게이트웨이 정보 수신 후 | 클라이언트 패킷 수신, 클라이언트에 패킷 송신 요청 |
| 채팅 서버               | 게임 서버          | 1 → M (모든쌍) | 채팅서버가 게임서버 정보 수신 후   | 게임서버로부터 채팅 요청 수신, 다른 게임서버로 채팅 브로드캐스팅 |

## 서버별 연결 책임 (Listener / Connector 명세)

각 서버가 어떤 포트를 열어 받고(Listener), 어떤 서버에 능동적으로 연결하는지(Connector) 명확히 합니다.


### 레지스트리 서버
- **InternalListener**: 모든 서버의 등록/폴링/하트비트 응답을 받는 포트
- **Connector**: 없음 (다른 서버에 능동적으로 connect하지 않음)
- **시작 순서**: 가장 먼저 시작되어야 하는 서버 (다른 서버들이 이 서버에 연결을 시도하기 때문)

### 로그인 서버
  **ClientListener**:
  - 클라이언트 로그인 요청 받는 포트
- **Connector**:
  - 레지스트리 서버에 connect (자기 등록 + 게이트웨이 정보 폴링)
  - 모든 게이트웨이 서버에 connect (인증토큰 사전 등록 용도)
- **시작 순서**: 레지스트리 서버 다음

### 게이트웨이 서버
- **InternalListener**:
  -  서버 내부 통신(게임서버, 로그인서버로부터)을 받는 포트
- **ClientListener**:
  - 클라이언트 접속 을 받는 포트
- **Connector**:
  - 레지스트리 서버에 connect
- **시작 순서**: 레지스트리 서버 다음 (게임서버보다 먼저 시작 권장)

### 게임 서버
- **InternalListener**:
  - 채팅 서버로부터의 연결을 받는 단일 포트
- **Connector**:
  - 레지스트리 서버에 connect
  - 모든 게이트웨이 서버에 connect
- **시작 순서**: 게이트웨이 서버 다음 권장 (게임서버가 게이트웨이에 connect해야 하므로)

### 채팅 서버
- **Listener**: 없음
- **Connector**:
  - 레지스트리 서버에 connect
  - 모든 게임 서버에 connect
- **시작 순서**: 게임 서버 다음

### 권장 시작 순서 요약
```
1. 레지스트리 서버
2. 로그인 서버, 게이트웨이 서버 (병렬 가능)
3. 게임 서버
4. 채팅 서버
```
> 단, 이 순서는 "정상 기동 시 권장" 일 뿐이며, 서버는 모두 **재연결을 무한히 시도**하는 구조이므로 어떤 순서로 시작해도 결국 정상 상태에 수렴합니다.

# 서버 스레드 구조
- 레지스트리 서버
	- N개의 IOCP Worker 스레드가 모든 요청을 처리합니다.
	- 1개의 RegistryServer 스레드가 서버와의 연결 상태를 체크합니다. 서버 연결이 끊겼는지를 지속적으로 체크합니다.

- 로그인 서버
	- N개의 IOCP Worker 스레드가 유저의 로그인을 처리합니다.
	- 1개의 RegistryClient 스레드가 레지스트리 서버로부터 게이트웨이 서버의 상태를 지속적으로 폴링합니다.
	
- 게이트웨이 서버
	- 1개의 Accepter 스레드에서 모든 클라이언트 연결을 처리합니다.
	- 1개의 RegistryClient 스레드가 레지스트리 서버로부터 게임서버의 상태를 지속적으로 폴링합니다.
	- N개의 IOCP Worker 스레드가 게임 서버로부터 받은 패킷을 대상 클라이언트에게 전달합니다.
	
	
- 게임 서버
	- N개의 IOCP Worker 스레드와 M개의 컨텐츠 스레드가 있습니다.
	- N개의 IOCP Worker 스레드는 패킷을 받아서 유저객체의 메시지 큐에 패킷을 삽입합니다.
	- 1개의 유저 객체는 자신만의 메시지 큐를 가집니다.
	- M개의 컨텐츠 스레드는 Stage객체(오픈월드필드, 던전)를 소유합니다.
	- 1개의 Stage객체는 유저 객체를 소유합니다.
	- 컨텐츠 스레드가 업데이트 될 때 자신 소유의 Stage객체를 업데이트 하고, Stage객체는 자신이 소유한 유저 객체를 업데이트 합니다. 이 때 유저객체의 메시지 큐의 패킷이 처리됩니다.
	
- 채팅 서버
	- N개의 IOCP Worker 스레드가 채팅 요청을 처리합니다.

# 서버 정보 전파 방식: Push + Pull 이중화

레지스트리 서버는 **두 가지 메커니즘**으로 서버 정보를 다른 서버에 전파합니다.

| 메커니즘                | 주체                  | 빈도            | 용도                              |
| ----------------------- | --------------------- | --------------- | --------------------------------- |
| **Push (이벤트 전파)**  | 레지스트리 → 다른 서버들 | 이벤트 발생 즉시 | 신규 등록, 연결 끊김, 종료대기 플래그 변경 등의 상태 변경을 즉시 알림 |
| **Pull (주기적 폴링)**  | 다른 서버 → 레지스트리   | 1분마다         | Push 누락 보완 + 레지스트리 재시작 시 자동 복구용 (fallback) |

### 이중화 이유
- **Push만 사용 시**: 네트워크 일시 단절로 Push가 누락되면 상태가 영구히 어긋날 수 있음
- **Pull만 사용 시**: 신규 게임서버 등록 후 최대 1분간 다른 서버가 그 존재를 모름 (실시간 확장 어려움)
- **둘 다 사용**: Push로 빠른 반영, Pull로 정합성 보장

### 폴링 대상

| 서버           | 폴링 대상              |
| -------------- | --------------------- |
| 로그인 서버    | 게이트웨이 서버 정보  |
| 게이트웨이 서버| 로그인 서버, 게임 서버 정보 |
| 게임 서버      | 게이트웨이 서버 정보  |
| 채팅 서버      | 게임 서버 정보        |

	
# 서버 연결 상태 감지 메커니즘

서버는 **3가지 채널**을 통해 다른 서버의 상태를 감지합니다.

## 1. 직접 TCP 연결 끊김 감지 (즉시)
- A 서버가 B 서버와 직접 TCP 연결을 맺고 있을 경우, B의 크래시/재시작/정상 종료는 **OS의 TCP 소켓 끊김 이벤트로 즉시 감지**됩니다.
- IOCP의 0바이트 수신 또는 에러 완료통지로 인지.
- 이 채널이 가장 빠르고 신뢰할 수 있음.

## 2. 레지스트리 서버를 통한 간접 감지 (수십 초 단위)
- A와 B가 직접 연결되어 있지 않은 경우 (예: 로그인서버는 게임서버와 직접 연결 없음).
- 레지스트리가 B의 상태 변경을 Push 또는 Pull로 다른 서버에 전파.
- 레지스트리는 하트비트(30초 주기, 1분 무응답 시 끊김 판단)로 B의 상태를 추적.

## 3. 어플리케이션 레벨 요청 실패 (즉시 ~ 수 초)
- 채팅서버처럼 게임서버에서 요청을 보낼 때 실패하면 즉시 끊김으로 판단하고 재연결 시도.

## 서버별 감지 매트릭스

"X 서버가 Y 서버의 끊김을 알아차리는 방법" 정리:

| 감지 주체 \ 끊긴 서버 | 레지스트리 | 로그인 | 게이트웨이      | 게임          | 채팅          |
| --------------------- | ---------- | ------ | --------------- | ------------- | ------------- |
| **레지스트리**        | -          | 직접 끊김 | 직접 끊김    | 직접 끊김     | 직접 끊김     |
| **로그인**            | 직접 끊김  | -      | 직접 끊김 + 레지스트리 | (직접 연결 없음) → 레지스트리 폴링 | (관심 없음)   |
| **게이트웨이**        | 직접 끊김  | 직접 끊김 + 레지스트리 | -    | 직접 끊김 + 레지스트리 | (관심 없음)   |
| **게임**              | 직접 끊김  | (관심 없음) | 직접 끊김 + 레지스트리 | -        | 직접 끊김 + 요청 실패 |
| **채팅**              | 직접 끊김  | (관심 없음) | (관심 없음) | 직접 끊김 + 레지스트리 | -        |

> "직접 끊김 + 레지스트리" 라고 쓴 칸은: 직접 TCP 연결로 즉시 감지하고, 레지스트리 통지로 한 번 더 확인(중복 안전장치)됨을 의미합니다.

# 서버 재시작/재연결 프로세스

## 재연결 주기 통일안

| 재연결 대상              | 재시도 주기 | 비고                            |
| ------------------------ | ----------- | ------------------------------- |
| 첫 부팅 시 레지스트리 연결 | **10초**  | 빠르게 부팅 완료하기 위함       |
| 운영 중 레지스트리 재연결  | **1분**   | 네트워크 폭주 방지              |
| 게임서버 ↔ 게이트웨이서버 | **1분**     |                                 |
| 로그인서버 → 게이트웨이서버 | **1분**   |                                 |
| 채팅서버 → 게임서버      | **1분**     |                                 |
| 게임서버 → 채팅서버      | **1분**     |                                 |

## 시나리오별 흐름

### 시나리오 A. 게임 서버가 새로 시작될 때
1. 게임 서버 부팅 → config 파일에서 자기 서버 ID 로드 → 즉시 사용 시작
2. 레지스트리에 connect 시도 (10초마다 재시도)
3. 연결 성공 → 자기 정보 등록 요청 (서버타입=Game, 서버ID, IP, Port)
4. 레지스트리가 ID 충돌 검증
   - **충돌 발견 시**: 등록 거부 → 게임서버는 에러 로그 남기고 종료 (운영자가 config 수정 필요)
   - **정상 시**: RegistryDB에 정보 저장 + 다른 서버에 Push 전파
5. 게이트웨이 서버들이 알림 수신 → 해당 게임 서버에 connect 시도
6. (병렬) 채팅 서버도 알림 수신 → 해당 게임 서버에 connect 시도
7. 게임 서버는 레지스트리에 게이트웨이 서버 정보 폴링 시작 (1분 주기)
8. 게임 서버는 정상 운영 상태

### 시나리오 B. 게임 서버가 크래시 났을 때
1. 게이트웨이 서버: TCP 끊김 즉시 감지 → 해당 게임서버에 접속 중이던 모든 클라이언트 연결 끊기
2. 채팅 서버: TCP 끊김 즉시 감지 → 해당 게임서버 제거 (브로드캐스트 대상에서 제외)
3. 레지스트리: 하트비트 1분 무응답 → 끊김 판정 → 다른 서버에 Push (중복 통지지만 안전장치)
4. 클라이언트는 재접속 시 다른 게임 서버로 라우팅됨

### 시나리오 C. 게임 서버가 크래시 후 재시작
- 시나리오 A와 동일하게 동작
- 동일한 config의 동일한 서버 ID로 다시 등록 → RegistryDB 검증 통과 (이전과 동일한 IP:Port:ID 매핑이므로)
- 다른 서버들의 캐시도 자연스럽게 갱신됨

### 시나리오 D. 레지스트리 서버 크래시
1. 모든 다른 서버: TCP 끊김 즉시 감지
2. 다른 서버들은 **자기가 가지고 있는 서버 정보 캐시**로 정상 운영 지속
3. 다른 서버들은 1분마다 레지스트리에 재연결 시도
4. 레지스트리 재기동 → RegistryDB에서 기존 정보 로드 → listen 시작
5. 다른 서버들이 재연결 성공 → 자기 정보 재등록 (검증 통과)
6. 정상 상태로 수렴

> **신규 서버 추가 제한**: 레지스트리가 죽어있는 동안 신규 서버가 부팅되면 레지스트리에 연결 시도만 반복하다가, 레지스트리가 살아나야 비로소 등록되고 운영 시작됩니다. 즉 **레지스트리 다운 = 신규 서버 추가 불가**. 이는 의도된 동작입니다.

### 시나리오 E. 게이트웨이 서버 크래시
1. 클라이언트들: 모두 연결 끊김 → 클라이언트가 재접속 시 다른 게이트웨이로 라우팅
2. 게임 서버: TCP 끊김 즉시 감지 → 해당 게이트웨이 서버에 연결되어 있던 모든 유저 객체 제거
3. 로그인 서버: TCP 끊김 즉시 감지 (로그인서버→게이트웨이 연결이 있으므로) → 해당 게이트웨이는 라우팅 후보에서 제외
4. 레지스트리: 하트비트로 인지 → 다른 서버에 Push (중복 통지지만 안전장치)

### 시나리오 F. 채팅 서버 크래시
1. 게임 서버: 채팅 요청 실패 → 채팅서버 연결 끊고 1분마다 재연결 시도
2. 레지스트리: 하트비트로 인지 → 다른 서버에 Push (게임서버가 이미 알고 있지만 안전장치)
3. 채팅 서버 재기동 시: 게임 서버들이 1분 내에 재연결 성공

### 시나리오 G. 로그인 서버 크래시
1. 클라이언트의 신규 로그인 불가 (기존 접속자는 영향 없음 — 게이트웨이/게임서버 통해 정상 플레이)
2. 게이트웨이 서버: TCP 끊김 감지 → 인증토큰 사전 등록 채널 끊김
3. 레지스트리: 하트비트로 인지 → 다른 서버에 Push
4. 로그인 서버 재기동 시: 게이트웨이에 다시 connect, 정상 운영


# Graceful Shutdown
- 게임서버, 게이트웨이 서버를 종료할 때
	- 레지스트리 서버에 게임서버 또는 게이트웨이 서버를 종료할 것임을 알립니다.
	- 레지스트리 서버는 해당 서버 정보에 종료대기 플래그를 설정합니다.
	- 종료대기 서버는 단지 유저가 이탈하기까지를 기다립니다. 나머지 로직들은 모두 정상적으로 작동합니다.
	- 종료대기 서버는 클라이언트가 더이상 접속할 수 없습니다.
		- 로그인 서버에서는 종료대기 게이트웨이서버의 IP는 클라이언트에게 전달하지 않습니다.
		- 게이트웨이서버는 종료대기 게임서버로 클라이언트를 입장시키지 않습니다.
	- 종료대기 서버에서 모든 유저가 이탈할때까지 기다린 다음, 모든 유저가 이탈하면 관리자가 수동으로 서버를 종료합니다.
	


# 클라이언트 로드밸런싱
로그인서버와 게이트웨이서버가 클라이언트 로드밸런싱을 담당합니다.  
로그인서버는 `클라이언트가 이전에 접속한 게이트웨이서버ID`를 관리합니다. 이 데이터 1건의 유효시간은 5분 입니다. 클라이언트가 빠르게 재접속 할때 이전에 접속했던 게이트웨이서버로 접속하게 해주는 용도입니다.  
게이트웨이서버는 `클라이언트가 이전에 접속한 게임서버ID`를 관리합니다. 이 데이터 1건의 유효시간은 5분 입니다. 클라이언트가 빠르게 재접속 할때 이전에 접속했던 게임서버로 접속하게 해주는 용도입니다.   

- 클라이언트가 로그인에 성공했을 때
	- 만약 `클라이언트가 이전에 접속한 게이트웨이서버ID`가 있다면 해당 게이트웨이서버의 IP:Port를 전달합니다.
	- 없다면 유저수가 가장 적은 게이트웨이서버의 IP:Port를 전달합니다.
	
- 클라이언트가 게이트웨이서버에 접속했을 때
	- 만약 `클라이언트가 이전에 접속한 게임서버ID`가 있다면 해당 게임서버에 클라이언트 접속을 알립니다.
	- 없다면 유저수가 가장 적은 게임서버에 클라이언트 접속을 알립니다.

# 로그인 및 인증
- 로그인
	- 클라이언트의 로그인은 ID, Password 검증 만으로 가능하게 합니다.
	
- 인증토큰
	- 클라이언트가 로그인서버에 로그인하면, 로그인서버는 인증토큰(8byte 랜덤 값)을 생성합니다. 인증토큰의 만료시간은 5분 입니다.
	- 로그인서버는 클라이언트에게 인증토큰과 게이트웨이서버 IP:Port를 전달합니다. 그리고 해당 게이트웨이 서버에게 클라이언트ID와 인증토큰, 만료시간을 전달합니다.
	- 클라이언트는 게이트웨이서버에 접속할 때 인증토큰을 전달합니다.
	- 게이트웨이서버는 클라이언트의 접속요청을 받으면 인증토큰과 만료시간을 검증하여 접속을 허가합니다.
	- 인증토큰 검증에 실패하면 클라이언트와의 연결을 끊습니다.
	

# 서버 ID 관리

## 서버 ID 발급 방식: Config 하드코딩 + 레지스트리 서버가 검증

서버 ID는 **각 서버의 config 파일에 하드코딩**하며, 레지스트리 서버는 **충돌 검증**만 담당합니다.

### 서버 ID 사전 할당 정책
서버 ID는 1~999 범위입니다. 운영 초기에 다음과 같이 ID 풀을 사전 할당합니다.

| 서버 종류        | 할당 ID 범위 | 비고                          |
| ---------------- | ------------ | ----------------------------- |
| 레지스트리 서버  | 1            | 1대만 존재                    |
| 로그인 서버      | 10           | 1대만 존재                    |
| 채팅 서버        | 20           | 1대만 존재                    |
| 게이트웨이 서버  | 100~199      | 최대 100대                    |
| 게임 서버        | 200~899      | 최대 700대                    |
| (예비)           | 900~999      | 향후 서버 종류 추가 시 사용   |

### 서버 등록 흐름
1. 서버가 부팅 → config 파일에서 자기 서버 ID 로드 → **즉시 사용 시작**
2. 서버가 레지스트리에 connect 시도
3. 연결 성공 시 자기 정보 등록 요청 (`서버타입`, `서버ID`, `IP`, `Port`)
4. 레지스트리가 RegistryDB에서 검증:
   - 동일한 `서버ID`가 다른 `IP:Port`로 이미 등록되어 있는가? → **있다면 등록 거부**
   - 동일한 `서버타입:IP:Port`가 다른 `서버ID`로 등록되어 있는가? → **있다면 등록 거부**
5. 등록 거부 시: 레지스트리가 거부 응답을 보내고, 서버는 에러 로그 남기고 **즉시 종료**
   - 운영자가 config의 ID 충돌을 인지하고 수정해야 함
6. 등록 수락 시: 레지스트리가 RegistryDB에 정보 저장 후 다른 서버에 Push 전파

> **참고**: RegistryDB에는 기존처럼 `서버타입:IP:Port ↔ 서버ID` 매핑이 저장됩니다. 이 매핑은 이제 ID 발급용이 아닌 **검증용 데이터**로 사용됩니다.

### 객체 ID 비트 구조
`[0(1bit)] [Timestamp(43bit)] [ServerID(10bit)] [Sequence(10bit)]`
데이터타입은 int64 입니다.

# 로그인서버
- 로그인한 모든 유저 관리map
	- 로그인에 성공한 유저ID, 유저가 접속한 게이트웨이서버 등의 정보를 관리합니다.
	- 만약 이미 로그인한 유저가 다시 로그인에 성공했다면, 유저가 접속한 게이트웨이 서버에 중복로그인을 알립니다. 그러면 게이트웨이 서버는 현재 유저의 연결을 끊습니다. 그리고 유저는 다시 게이트웨이서버에 접속합니다.

# 게이트웨이 서버
- 유저 클래스
	- 1개 유저의 세션을 관리하는 클래스 입니다.
	- 세션(TCP소켓, IP, Port), 연결된게임서버ID 등의 데이터를 가집니다.
	
- 유저 관리용 map
	- Key=유저ID, Value=유저소켓

- 클라이언트가 게이트웨이서버에 접속하여 인증토큰 검증까지 마치면 유저객체를 생성하고 map에 삽입합니다.
- 클라이언트와의 연결이 끊기면 map에서 제거하고 게임서버에 연결끊김을 알립니다.
- 게임서버가 죽으면 해당 게임서버에 접속해있던 모든 클라이언트의 연결을 끊습니다.
